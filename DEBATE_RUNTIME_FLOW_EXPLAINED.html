Council 辯論 Runtime 現況圖

這份圖對應目前實作中的 src/lib/council.ts 和 src/hooks/use-council-review.ts。重點不是產品理想流程，而是現在後端真的怎麼跑、 哪些分支是特殊情況、以及我剛剛說要補測試時到底是在補哪幾條分支。

這四個分支現在是在說什麼

resume

Resume 不是重新開始，而是接著既有 session 狀態跑

runCouncilSession(..., { resume: true }) 時，後端會先讀 existingTurns 和 existingConclusion。 已經完成的 round / seat 會跳過，只補尚未完成的部分。

partial round 2

Partial Round 2 指的是 Round 2 跑到一半中斷

例如 5 個 seats 只完成 2 個就斷線。現在重跑時不會重新做 divergence classifier， 而是直接把已存在的 Round 2 turns 當作 round2TurnsSoFar， 讓後面 seats 接著跑。

round2_skipped

Round 2 skipped 是「後端判定沒必要進第二輪」

Round 1 跑完後會做 divergence classification。如果 level 是 none 或 low，就直接 emit round2_skipped，接 moderator，不再讓 seats 交叉辯論。

high_divergence_warning

High divergence warning 是提醒 UI：共識可能是被 moderator 硬合成的

如果 divergence level 是 high，Round 2 還是會照跑， 但在進 moderator 前會多 emit 一個警告事件，提示使用者要特別看 dissent / forced consensus 的風險。

所以「要改什麼」其實是在補哪裡

Resume / partial Round 2
這不是我打算改產品邏輯，而是要補測試保護現在這條邏輯。 目前後端已經有 resume 行為，但還缺 source-level integration test 去驗證： 已完成的 seats 會不會被跳過、後續 seats 會不會拿到正確的 round2TurnsSoFar。
round2_skipped
目前後端已經會 emit 這個事件，前端也會收。 要補的是測試，確保當 divergence = none / low 時，真的不會偷偷跑 Round 2， 而是直接進 moderator。
high_divergence_warning
目前它只是警告事件，不會改變主流程。 要補的是測試，確認 divergence = high 時，Round 2 照跑、warning 會出現、 moderator 還是照常執行，這三件事會同時成立。
