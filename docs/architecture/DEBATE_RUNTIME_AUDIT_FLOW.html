Council Runtime Audit

Debate 流程 / RAG 次數 / 搜尋次數 / Citation 顯示

依實際程式碼整理。主路徑來自 session-orchestrator.ts → turn-executor.ts → agentic-runtime.ts → rag.ts / web.ts → evidence-annotations.ts。

核心上限

Round 1 並行度預設 2
Round 2 依序執行
每席每回合 runtime tool 上限 4 次
每席每回合可先被強制 preload 1 次 RAG
學術 review 預設工具

大多數 reviewer 只有 rag_query。 novelty / citations 類 reviewer 會再加 search_papers。

預設 academic debate 幾乎不會直接用 web_search / fetch_url。
RAG 在 debate 裡是 extractive 模式

turn-executor.ts 會把 seat runtime 的 rag_query 強制覆寫為 answer_mode=extractive。

所以 debate 內的 RAG 不會再多叫一個回答模型做 synthesize，主要是 DB 檢索 + 摘要片段。
外網搜尋的真實定義

search_papers 是查 OpenAlex / arXiv / Semantic Scholar。這算外部網路查詢，但不是一般網頁搜尋。

若 source=both，一次 search_papers 會打 2 個 provider: OpenAlex + arXiv。

1. Session 啟動

POST /api/sessions/[id]/run

SSE 開流，進 runCouncilSession()。先檢查 owner、quota、resume / restart、stale session。

↓
Round 1 建 prompt

每個 seat 收到固定 Markdown 架構：Position、Key Assumptions、Main Risks、Strongest Counterargument、可選 Evidence。

來源: buildRound1Prompt()

2. Seat 進場前的強制 RAG

Preload paper evidence

若 seat 有 library_id 且可用 rag_query，先自動打一個 rag_query(limit=5, tag=council:lib:...)。

這一步每席每回合最多 1 次。
↓
結果先塞回 prompt

以 [TOOL_RESULT tool="rag_query"]...[/TOOL_RESULT] 形式注入，並要求 reviewer 最終文字至少引用其中一個 marker / title / finding。

3. Seat runtime 回圈

runAgenticRuntime()

每席每回合最多跑 4 輪工具呼叫，工具總數上限 4。支援的 safe tools 含 rag_query、search_papers、web_search、fetch_url 等。

↓
academic 預設通常只會用

RAGsearch_papers

review preset 沒把 web_search 放進大多數 academic seat，因此一般 debate 不太會查 Brave / DuckDuckGo。

↓
工具結果先存 evidence

每次 tool_call / tool_result 都會寫入 council_evidence，並抽出 sourceRefs 給前端與 moderator 使用。

4. Round 2 條件分支

先做 divergence check

Round 1 結束後跑 classifyDivergence()。若分歧不足，直接跳過 Round 2。

↓
Round 2 是依序，不並行

每個 seat 都能看到先前的 Round 2 發言後再回應，輸出結構改成 Challenge + Stance + 可選 Evidence。

5. Moderator 與輸出

Moderator 不再查資料

Moderator 直接吃所有 turn 與 evidence count，不跑 seat tool loop。

↓
輸出固定 JSON

summary、consensus、dissent、action_items、veto、confidence、confidence_reason

若第一次沒吐合法 JSON，會做 1 次嚴格 JSON retry。
↓
前端顯示

左側是各 seat 的 Markdown 訊息，右側 Source Panel 顯示所有 sourceRefs。文內 hover / underline citation 由 evidence-annotations.ts 做配對。

6. Citation 真實性

會顯示來源，但不是嚴格逐字原文保證

RAG snippet 來自 chunk，經過 compactSnippet() / cleanSnippet()，會截短、壓空白，所以常是「整理過的片段」不是完整 verbatim 原句。

↓
顯示邏輯有兩層

1. 若文字裡有 [1] 這種 marker，直接對應 source。 2. 若沒有 marker，前端會用 snippet / title token 做 heuristic 對句子配對。

所以 citation UI 通常夠用，但不是論文編輯器級的精準對齊。
每席每回合 RAG
1 + 4

若有 library_id，先 preload 1 次；runtime 裡最多再叫 4 次工具，所以理論上最多 5 次 RAG。

每席每回合外部查詢
0 - 4

academic 預設通常是 search_papers，不是 web_search。沒配置該工具就會是 0。

一次 search_papers
1 - 2 API

source=both 時實際會查 OpenAlex + arXiv；若指定 semantic_scholar 則只打一個 provider。

Moderator retry
0 - 1

只在 JSON 不合法時補做 1 次非串流 retry，目的是把 synthesis 轉成固定 JSON 結構。

結論: 這個 debate runtime 的主體不是「瘋狂上網搜資料」，而是「先用 session library 做受限 RAG，再讓少數 reviewer 補查 academic paper search」。 citation 會帶出來源標題、URL、snippet，通常能看出依據；但 snippet 不是保證逐字原文，且文內高亮在沒有 marker 時是 heuristic 對位，不應視為 100% 精準逐句引用。
